Multi-tenant HRMS · layered view

Four layers. Tenants differ in one.

Read it top to bottom. A request flows down; configuration is data the engines read. The whole trick of multi-tenancy: every company shares the same engines and the same schema — they only differ at the config layer.

You originally pictured a “logic GUI layer.” There isn't one. The GUI just edits layer 02 (config data); the logic lives in layer 03 (the engines). That single correction is what lets one codebase serve every tenant — change the config, not the code.

What goes where

The line that keeps the platform sane: configure freely, extend carefully, and never let the critical stuff become config.

Configure
Layer 02 · data

Lookups, approver chains, templates, reminders, document rules, field visibility, leave policies.

Extend
Plugin · defined hooks

Tenant-specific behavior at versioned extension points — sandboxed, not free-form config.

Never config
Layer 03 / 04 · code

Payroll math, audit trail, tenant isolation. Configure the inputs — never the algorithm.
